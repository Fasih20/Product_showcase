# ADR-003: Admin API Authentication Strategy

Status: proposed

## Context
The project requires a new `/api/admin` endpoint to return system status. This endpoint must be secured to prevent unauthorized access. Existing ADR-002 addresses user SSO for the main application, but a dedicated, simpler authentication mechanism is needed for internal admin API routes, which may not involve a full UI-based SSO flow.

## Decision
For the initial implementation of internal admin API routes, we will use a simple API key validation. The API key will be passed via an `X-API-Key` HTTP header and validated against a securely stored environment variable (`ADMIN_API_KEY`). This provides a quick and effective security layer for internal tools.

## Consequences
- Requires careful management and rotation of the `ADMIN_API_KEY` environment variable.
- Not suitable for public-facing APIs or scenarios requiring fine-grained access control beyond a single key.
- May need to be replaced or augmented with a more robust solution (e.g., OAuth2, JWTs, or integration with a dedicated admin authentication service) if a complex admin UI is developed or if multiple admin roles are introduced.
- Simplifies initial development and deployment for internal admin endpoints.
